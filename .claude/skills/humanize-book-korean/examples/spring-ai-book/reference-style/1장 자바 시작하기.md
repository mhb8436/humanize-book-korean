1. 자바 시작하기

자바(Java)는 프로그래밍 언어의 한 종류이고 또한 플랫폼을 지칭하기도 한다.  프로그래밍언어를 분류하는 방식에는 여러가지가 있다. 첫번째로 프로그래밍언어는 기계가 인식하기 쉽고 사람이 이해하기 어려운 어셈블리어와 같은 저-수준의; 언어에서 부터 사람이 이해하기 쉬운 고-수준의 언어가 있다.  C, C++ 그리고 자바와 같은 언어는 고-수준의 언어이다.  두번째로   포트란(Fortran)이나 C와 같은 절차형 언어, 그리고 자바(Java), Object C, C++,  스몰톡(SmallTalk)와 같은 객체지향 언어로 나뉠 수 있다. 절차형 언어는  프로그램의 순서 대로 위에서 아래로 읽어나가면서 실행되는 언어를 말하고 객체지향언어는 객체가 실행의 단위가 되는 언어를 의미한다.  우리는 이 책에서 객체지향 언어인 자바의 특성을 이해하고 사용하는 방법과 개념을 완벽하게 학습할 것이다.

1.1 자바 이야기
자바는 이 기종 환경, 네트워크 분산 환경에서 애플리케이션을 만들고 구동시키기에 적합하도록 설계되었다. 자바로 만들어진 프로그램은 운영체제(OS)의 특성에 관계 없이 구동 된다. 즉 우리가 일반적으로 사용하는 윈도우(Window)나 Mac의 OSX 운영체제 그리고  리눅스(Linux)나 유닉스(Unix)에서도 구동이 된다. 최초 자바 프로젝트에서 프로그래밍로서의 자바는 다양한 네트워크 장치들과 임베디드 시스템들에서도 잘 구동될 수 있도록 설계 되었다. 초기 자바의 설계목표는 소형이고, 신뢰성 있으며, 분산환경 하에서 구동 될 수 있는 언어를 만드는 것이었다. 초기에 자바는 C++을 기반으로 만드려고 했다가 C++언어는 문법과 개념이 어려워 이를 포기하였고, 자바 프로젝트 팀은 새로운 플랫폼을 만들기로 하였다. 이 새로운 플랫폼은 Eiffel, SmallTalk, Object C, Cedar/Mesa와 같은 다양한 언어를 참조하였다.  이렇게 새롭게 만들어진 플랫폼 언어는 분산 네트워크 기반의 사용자 애플리케이션에 적합한 것으로 판명 되었다. 특히 자바언어는  WWW(World Wide Web) 기반의 시스템과 네트워크 임베디드 디바이스에서 각광을 받고 있다.

1.1.1 자바의 탄생/역사
자바는 1995년에 썬마이크로 시스템즈(지금은 오라클에 인수됨)의 제임스 고슬링(James Gosling)에 의해 만들어졌다. 객체 지향 프로그래밍 언어라고 불리고 있고 오라클에서 무료로 제공하고 있다. 자바는 1991년에 썬마이크로 시스템즈에서 그린 프로젝트(Green Project)라는 이름으로 처음 시작되었다. 프로젝트의 목표는 임베디드 시스템과 분산 네트워크 환경에서 구동하기 위한 프로그래밍언어와 플랫폼을 만드는 것이었다.
자바는 현재 웹어플리케이션 분야에서 전세계에서 가장 많이 쓰이는 언어 중에 한 가지이고 우리에게 익숙한 스마트폰 운영체제인 안드로이드 OS에서도 자바를 기반 으로 구동 되고 있다.  책을 쓰는 시점에서는 현재 자바 7까지 나와있고 보안 문제가 있어서 조만간 8 릴리즈가 출시될 것으로 기대되고 있다.

제임스 고슬링(James Gosling) 은 1991년 프로젝트를 위해 자바언어를 만들기로 하였다. 이 프로젝트는 그린(Green), 오크(Oak) 프로젝트로도 불렸으나 결국 자바(Java)라는 이름으로 불리게 된다. 이 프로젝트의 목표는 이 기종  OS 간에도 실행가능하고 분산,네트워크 환경에서도 안정적으로 수행되는 플랫폼과 프로그래밍 언어를 만드는 것이었다. 1995년에 프로젝트가 종료되고 나온 결과물은 자바1.0 이었다. 자바는 “Write Once, Run Anywhere”(한번 쓰고 여러 곳 에 서 실행한다)를 보장하였고 상당히 안정적인 플랫폼을 제공하였고 꽤 좋은 보안 설정을 통해서 파일 또는 네트워크 자원에 대한 접근을 제어할 수 있었다. 자바가 출시 될 당시 자바로 만들어진 애플릿(Applet)은 거의 모든 브라우저에서 지원하였다. 애플릿(Applet)은 브라우저에서 제공하지 못하는 세션 유지 또는 브라우저의 HTML보다 화려한 UI를 제공하였고 많은 인기를 끌었다. 2000년 이후에 JDK 1.2부터 출시되었고 다양한 플랫폼에서 제공할 수 있도록  여러가지 스펙의 자바 버전들이 만들어 졌다. 예를 들어 J2EE는 엔터프라이즈용 애플리케이션을 실행할 수 있었고 J2ME는 모바일 애플리케이션을 실행할 수 있었다. J2SE는 스탠다드 에디션으로 일반적인 PC 환경에서 애플리케이션을 실행할 수 있었다. J2SE는 Java의 가장 기본적인 에디션이고 우리가 필요한 모든 문법들을 포함하고 있다. J2EE와 J2ME는 J2SE를 기반으로 엔터프라이즈 환경과 모바일 환경에 필요한 기타 스펙들이 빠지고 추가된 버전이다.  J2EE, J2SE, J2ME는 2006년 이후 Java EE, Java SE, Java ME로 이름이 바뀌었다.

자바 플랫폼, 엔터프라이즈 에디션(Java Platform, Enterprise Edition: Java EE)(주1)는 자바를 이용해서 서버 환경에서 개발하기 위한 플랫폼이다.  Java EE  플랫폼은 자바 플랫폼, 스탠다드 에디션에 웹 애플리케이션 서버에서 동작하는 장애복구 및 분산티어를 제공하는 자바 소프트웨어의 기능을 추가한 서버를 위한 스펙이다.

자바 플랫폼, 스탠다드 에디션(Java Platform, Standard Edition, Java SE)(주2)는 데스크톱 및 서버, 고사양의 임베디드 시스템에서 구동되는 표준 자바 플랫폼으로 표준 컴퓨팅 환경을 위한 자바 가상 머신의 규격과 API이다. Java EE, Java ME등은 그 목적에 따라 Java SE를 기반으로 API  추가하거나 API 중 일부를 택하여 정의한다.

자바 플랫폼, 마이크로 에디션(Java Platform, Micro Edition)은 Java ME, J2ME(주3) 등으로 불린다. 이는 한정된 자원을 가진 PDA, 휴대전화, 셋톱박스 등에서 JAVA 프로그래밍 언어를 지원하기 위해 만들어진 플랫폼 중 하나이다.

자바는 썬 마이크로시스템즈에서 만들었고 1997년 ISO/IEC에 표준화를 하려고 하였으나 표준화 되지 못하였고 사실상의 산업계 표준(de facto) 표준이 되었고 자바 커뮤니티 프로세스(Java Community Process)에서 관리하고 있다. 썬 마이크로시스템즈는 자바의 대부분을 무료로 제공하였으나 일부는 유료로 제공하였다. Java EE의 경우 일부는 유료로 라이센스를 제공하였다. 2007년에 썬 마이크로시스템즈는 대부분의 자바를 GPL 라이센스로 소스를 오픈하였다.  2009년 썬 마이크로시스템즈를 오라클이 인수하였고 자바에 대한 권리 및 유지보수 또한 오라클로 넘어가게 되었다.

1.1.2 자바의 개정현황
자바 플랫폼에서 Java SE, Java EE, Java ME와 같이 에디션 분류가 도입된 것은 JDK 1.2 버전인 Java 2 부터 이다. JDK 1.2 부터는 이전의 1.1.x  버전과 차이를 감안하여 자바 2(Java 2)라고 불리우게 되었다. 그리고 엔터프라이즈 에디션(자바 2 EE)에서는 서버 시스템을 위한 기능들이 추가되었고,  자바 임베디드 에디션(자바 2 ME)를 추가하였다. 자바 임베디드 에디션은 그 전의 퍼스널자바(PersonalJava)와 임베디드자바(EmbededJava)를 계승하였다. 1.2.x 버전, 1.3.x 버전, 1.4.x 버전 까지 Java SE 1.4.x 또는 J2SE 1.4.x  식으로 명명하다가 1.5 버전에서 부터는  J2SE 5.0 식으로 바뀌었고 1.6 버전 부터는 자바 SE 6 형태로 다시 명명하게 되었다. 하지만 내부적으로 1.x.x 형태로 계속 유지되고 있다.  각 버전 별로 달라진 이력을 살펴보자.  이 부분을 내용이 조금 어렵고, 모두 이해할 필요는 없지만, 우리가 사용 해야하는 자바의 주요 기능들이 어떻게 변해왔는지 알아두면 자바를 학습 하는데에 좀 더 도움이 되기 때문에 간략하게 살펴보고 넘어가도록 하자.

1995년 자바 알파, 베타 버전
이 당시의 버전의 API(Application Prcoess Interface)와 ABI(Application Binary Interface)는 불안정한 상태로 공개 되었다.  WebRunner라는 자바 웹 브라우저가 기본적으로 제공되었다.

1996년 JDK 1.0
오크(Oak)라는 코드명으로 불리었던 초기 버전이다. 최초로 가장 안정된 버전은 Java 1 이라고 불리는 JDK 1.0.2 버전 이었다.  그전까지의 JDK 1.0.1 버전은 접근 제어자 private와  protected 가 구분되지 않았다.

1997년 JDK 1.1
이 당시의 버전 부터 자바 언어에 많은 기능 들이 추가 되기 시작하였다. 가장 큰 변화로는 다음의 기능이 추가된 것이다.
-  이너 클래스(inner class) : 객체 지향 프로그램에서 클래스 내에 클래스를 선언하는 구조
-  자바빈즈(JavaBeans) : 자바에서 재사용 가능한 컴포넌트, 다양한 오브젝트를 단일 클래스에 구조화 저장할 수 있다.
- JDBC :  Java-based data access 의 약자. 이 API는 데이터베이스에 접근하기 위한 클라이언트 프로그램의 기능을 포함하고 있다.
- RMI : Java Remote Method Invocation의 약자. 원격에서 자바 메소드를  호출하는 방법을 정의한다.
- reflection : 프로그램 내에서 프로그램 실행 중에 객체의 구조와 행동을 조사하거나 변경하는 능력을 말함

1998년 J2SE 1.2
이 때의 코드명은 Playground라고 불리었다. 이후의 Java 릴리즈에서  J2SE 5.0 까지는  Java 2 라는 이름으로 불리웠다.  버전명은 J2SE(Java 2 Platform Standard Edition), J2EE(Java 2 Enterprise Edition), J2ME(Java 2 Micro Edition) 으로 불리웠다. 이 버전에 추가된 기능은 다음과 같다.
- Swing API : 자바의  GUI 위젯 툴킷.
- JIT compiler : just-in-time의 약자. 자바와 같은 버추얼 머신 상에서 구동 되는 컴퓨터 프로그램의 성능을 향상 시켜주는 방법
- Java IDL : CORBA에서 사용하는  IDL 언어의 자바 버전
- Collection framework : List, Map, Hash와 같은 데이터 구조를 다루는 자바에서 제공하는 기본 프레임워크

2000년 J2SE 1.3
이 버전은 코드네임은 Kestrel 로 불리었다. 주요한 변경 이력은 다음과 같다.
- HotSpot JVM의 포함
- JDNI 추가 : Java Naming and Directory Interface 의 약자. 이름으로 자바 객체와 데이터를 조회할 수 있는 자바 API와 디렉토리 서비스 이다.
- JPDA 추가 : Java Platform Debugger Architecture의 약자. 자바 디버깅 코드를 위한 모음이다.

2002년  J2SE 1.4
코드네임은 Merlin 이었고 자바 커뮤니티 프로세스(Java Community Process) 주도 하에 개발된 첫 번째 릴리즈이다. 주요 변경 이력은 다음과 같다.
- assert 키워드 추가
- 펄(Perl)기반의 정규식(regular expressions) 수용할 수 있도록 추가
- IPv6를 지원
- 로깅 API
-  XML 파서의 내장
- 보안, 암화화 기능 확장
-  Java Web Start 포함
- Preference API 추가

2004년 J2SE 5.0
이 버전의 코드네임은 Tiger 였고 다수의 언어적 기능이 추가 되었다. 1.4 버전과 마찬가지로 자바 커뮤니티 프로세스 주도 하에 변경되었다.
- 제네릭(Generics)  : 콜렉션에 대해 컴파일 시점에 타입의 안정성을 확보. 제네릭을 쓰면 타입 캐스팅이 별도로 필요 없다.
- 어노테이션(annotation) 기반 메타데이터(metadata) : 클래스나 메소드에 첨가 되어야할 추가 적인 정보를 기입.
- 오토박싱/언박싱(autoboxing/unboxing) : 프리미티브 타입(int 형)과 프리미티브 래퍼 클래스(예: Integer)간의 자동 형변환
- Enumeration   enum은 타입에 안전한 값들의 목록을 제공한다. 예를 들어 Day.MONDAY, Day.TUESDAY, Day.WENDESDAY 등올 사용한다.
- 확장된 for each 문 : 기존의 for 문 문법보다 배열 또는 Collection을 반복시키기 쉬운 문법 제공

이외에도 J2SE 5.0에서는 많은 변경사항들이 있다.  자세한 변경 사항들을 위키(주1)을 참고하여도 좋다.

2006년 Java SE 6.0
Mustang이란 코드명을 사용하였다. 이 버전 부터 썬 마이크로시스템즈에서는 J2SE라는 이름을 버리고 Java SE라는 이름을 사용하였다.  6.0에서도 1.4에서 5.0으로 변경될 때 만큼의 많은 추가가 이루어지지는 않았지만 다수의 기능개선이 이루어 졌다.
- 스크립트 언어 지원 기능 강화
- 플랫폼과 스윙(Swing)에 대한 성능 개선
- JAX-WS(주2)을 통한 웹 서비스 지원 강화
- JDBC 4.0 추가
- StAX 파서(주3)를 포함한 JAXB 2.0 업그레이드
- 스윙(Swing)컴포넌트의 기능 개선
- JVM(Java Virtual Machine)의 성능 향상

2011년 Java SE 7
Java SE 7의 코드명은 Dolphin 이다. 2011년 7월 부터 단계적으로 릴리즈 되고 있다. 책을 쓰는 시점의 가장 최신 버전이다.  현시점에 보안 문제가 발생하여 8 버전의 릴리즈가 거론되고 있다.  Java SE 7의 가장 큰 기능 추가는 다음과 같다.
- swtich 문에서 String 객체를 사용할 수 있다
- try 문에서 자동 리소스 관리를 추가하였다.
- 타입 체크를 강화하여 여러 개의 예외를 던지거나 잡을 수 있다.

이 외에도 많은 업데이트가 이루어졌다. 자세한 내용을 더 알고 싶으면 http://en.wikipedia.org/wiki/Java_version_history 문서를 참고하길 바란다.

지금까지 개정된 자바 이력에 대하여 살펴봤다.  자바의 개정 이력은 초심자들이 보기에 어려운 면이 있기 때문에 이 책을 다 공부하고 나서 살펴보아도 좋다. 자바는 자바 커뮤니티 프로세스에 의해서 지속적으로 업데이트 되고 있고 다른 언어의 좋은 기능들을 지속적으로 추가하고 있기 때문에 버전이 업데이트 될 때 마다 지속적으로 학습이 필요하다. 이 책을 모두 공부하고 난 뒤 자바로 계속 개발을 하고자 한다면 http://docs.oracle.com/javase/ 을 방문하여 꾸준한 재학습을 해야 버전에 맞는 양질의 코드를 만들어낼 수 있다. 책을 쓰는 현재 시점에서 현장에서 가장 많이 사용되는 자바 버전은 Java SE 1.6 이다. 이 버전은 다수의 현장에서 검증된 가장 안정적인 버전이며 이 버전은 시간이 지남에 따라 계속 업그레이드 되고 있다.


1.1.3 자바의 특징
지금까지 자바의 만들어진 배경과 변경된 이력을 살펴보았다. 자바가 만들어진 배경에는 몇 가지 철학이 존재한다. 자바는 이 철학을 바탕으로 몇 가지 주된 특징을 가지고 있다. 자바를 만들 때 기본적인 사상은 첫 번째 간단하고, 객체지향 개념을 사용해야  하고, 두번째 안전하고 견고해야 하며, 세번째 아키텍처 중립적이고 휴대성이 있어야 한고 네번째 고성능이어야 하며, 다섯번째 인터프리터 언어이고, 쓰레드를 사용해야 하며 동적이어야 한다라는 다섯가지 철학을 바탕으로 하고 있다.
자바 탄생의 첫번째 철학은 “간단하고, 객체지향 개념을 사용해야 한다.” 이다. 여기서 간단하다라는 의미는 C나 C++과 달리 간단한 구조로 프로그램을 만들어야 한다라는 의미로 받아들이면 된다. 객체지향개념은 6장 에서 자세히 다루겠지만 Object-C, SmallTalk 등과 같은 객체지향언어의 특성을 가져야 함을 의미한다. 즉 첫번째 법칙은 자바는 객체지향개념을 가지고 간단하기 프로그램을 만들수 있어야 함을 의미한다
두번째 철학은 “안전하고 견고해야 한다” 이다. 여기서 안전하다는 말은 JVM자체에 바이러스를 막고 템퍼링(tampering)과 같은 공격에 대한 방어 체계가 있기 때문에 안전하다는 말로 이해하면 될 것 같다. 또한 견고해야 한다는 의미는 자바에는 C나 C++과는 달리 포인터에 대한 개념이 없고 메모리에 대한 관리를 JVM이 직접 해주기 때문에 견고하다는 의미로 이해하면 될 것 같다.
세번째 철학은 “아키텍쳐 중립적이고 휴대성이 이어야 한다” 라는 의미는 자바로 만들어진 프로그램은 Windows, Linux, Mac OSX, Unix와 같은 모든 운영체제에서 재빌드나 재컴파일 없이도 구동되어야 함을 의미한다. C나 C++과 같은 경우는 해당 운영체제에 맞게 재빌드를 해야한다.
네번째 철학은 “고성능이어야 한다” 이다. 고성능의 의미는 자바로 만들어진 응용프로그램을 수행함에 있어서 JVM은 최적의 환경으로 수행될 수 있도록 보장한다. 예를들어 메모리의 경우 JVM은 가비지 콜렉터를 자동적으로 수행 해줌으로써 자바로 작성된 응용프로그램이 최대로 메모리를 사용할 수 있는 환경을 만들어 준다.
다섯번째 철학은 “인터프리터 언어이고, 쓰레드를 사용해야 하며 동적이어야 한다”이다. 인터프리터 언어라는 것은 자바는 컴파일하고 바로 실행할 수 있기 때문에 개발주기가 빨라지는 것을 의미한다. 쓰레드는 13장에서 자세히 배우겠지만 자바는 프로세스 내에서 여러 개의 쓰레드를 이용하여 동시에 처리할 수 있다. 쓰레드를 이용하면 처리 속도와 처리량이 늘어나게 된다.

자바언어의 철학은 바로 자바언어의 특징이기도 하다. 지금부터는 자바언어와 자바 플랫폼에 대해서 좀 더 자세하게 실펴보자.
자바 프로그래밍언어에서 모든 소스코드는 일반 텍스트 파일이다. 이 파일의 확장자는 .java 로 끝나게 된다. .java확장자로 끝나는 소스코드는 컴파일이 되면 .class 확장자를 가진 파일이 만들어진다. 컴파일은 javac 라는 컴파일러를 통해서 이루어진다. .컴파일을 통해서 생성된 .class 파일은 운영체제에 특정된 코드를 가지고 있지는 않는다. 이는 바이트코드(bytecode)라고 부르는 JVM(Java Virtual Machine) 에서 필요한 중간 언어이다.  java 라는 실행 툴로 .class 파일을 실행하게 되면 JVM  인스턴스 한개를 가지고 프로그램이 실행된다.  이 내용은 그림 1.1에 자세히 설명되어 있다.


[그림 1.1] 자바 프로그램 개발 절차의 요약


Java VM(Java Virtual Machine)은 다양한 운영체제를 지원하기 때문에 동일한 .class 파일은 마이크로소프트 윈도우나, 리눅스, 유닉스, Mac OSX와 같은 여러 운영체제에서 실행될 수 있다. 일부 Java VM의 경우에는 성능 향상을 위해서 실행 중에 운영환경에 맞게 재컴파일하거나 병목을 찾는 작업과 같은 추가적인 단계를 더 거치기도 한다.  그림 1.2는 동일한 자바 프로그램이 여러 개의 운영체제에서 구동되는 모습을 보여주고 있다.

[그림 1.2] Java VM을 통해서 하나의 자바 프로그램이 여러개의 운영체제에서 실행될 수 있다.



앞서 설명한 바와 같이 자바는 언어이기도 하고 플랫폼이기도 하다. Java VM(Virtual Machine)은 자바 언어로 작성된 파일을 실행할 수 있는 플랫폼이다. 플랫폼이란 의미는 프로그램을 실행할 수 있는 하드웨어 또는 소프트웨어 환경을 말한다. 마이크로소프트 윈도우(Window)나 리눅스, 솔라리스, HP-UX, AIX와 같은 유닉스(UNIX)와 같은 운영체제도 플랫폼의 일종이다. 대부분의 플랫폼은 하드웨어와 운영체제의 조합으로서 설명된다. 하지만 자바의 경우 플랫폼의 의미는 조금 다르다.  자바 플랫폼은 두개의 컴포넌트로 구성이 되어있다. 자바 가상 머신(Java Virtual Machine)과 자바 애플리케이션 프로그래밍 인터페이스(Java Application Programming Interface:API) 로 구성된다. 자바 가상 머신(Java Virtual Machine)은 다양한 운영체제와 하드웨어에 적합한 여러 개의 버전을 제공하고 있다. 자바 가상 머신은 중간 언어인 바이트코드를 해석해서 실행하는 역할을 한다. 앞서 언급된 JVM, Java Virtual Machine, Java VM 등의 용어는 바로 자바 가상 머신을 의미한다. API는 개발자가 프로그램을 만들 때 이용할 수 있는 미리 만들어진 모음을 의미한다. API는 클래스와 인터페이스들로 구성된 라이브러리(Library)라고 할 수 있다. 이 라이브러리들은 패키지(Package)라고 하기도 한다. 우리가 만들 자바 프로그램은 그림 1.3에서와 같이 API와 자바 가상 머신으로 구성된 자바 플랫폼 위에서 구동된다.


[그림 1.3] API와 자바 가상머신은 우리가 만든 자바프로그램과 하드웨어와 운영체제 사이에 존재한다.


플랫폼에 독립적으로 구동될 수 있는 자바 플랫폼은 운영체제에 특화된 네이티브(native) 코드보다 조금 느릴 수 있지만 컴파일러와 가상 머신 기술들로 네이티브 코드에 근접할 만큼의 성능을 갖출 수 있다.

지금까지 자바의 특징에 대해서 간략하게  살펴보았다.  자바는 객체지향 프로그램 언어임과 동시에 플랫폼을 의미하기도 한다.  그럼 이제 부터는 이러한 자바가 과연 실생활에서 어떤 모습으로 사용되고 있는지 알아보자.

1.2		자바로할 수 있는 일
실 생활에서 자바로 만들어진 제품을 떠올린다면 대표적으로 웹 페이지와 스마트폰 운영체제인 안드로이드일 것이다. 자바가 탄생하고 나서 가장 각광을 받은 부분이 JSP/Servlet을 포함한 J2EE일 것이다. 인터넷이 퍼지기 시작하면서 웹페이지, 웹시스템, 웹애플리케이션에 대한 수요가 폭발적으로 증가하였고 자바 플랫폼은 플랫폼 독립적이라는 장점을 바탕으로 많이 사용되게 되었다.  웹페이지, 웹시스템, 웹 어플리케이션의 의미는  구글, 네이버와 같은 웹기반으로 운영되는 시스템, 사이트에서 구동되는 애플리케이션이라고 생각하면 될 것 같다.
아이폰이 등장한 이후로 안드로이드 스마트폰은 아이폰에 대항할 수 있는 유일한 운영체제로 독보적으로 사용되고 있으며 구글에서 관리 하면서 오픈소스로 유지되고 있다. 오픈소스는 소스를 공개해서 누구나 라이센스에 대한 제한 없이  사용할 수 있는 소프트웨어를 말한다. 안드로이드는 자바 가상 머신(Java VM)기반으로 구동되며, 자바 언어로 작성된다.  안드로이드 스마트폰에서 볼 수 있는 모든 앱들은 자바언어로 만들어져 있다. 아이폰은 Object C 라는 언어로 만들어져 있다. 아이폰 뿐만 아니라 맥용 애플리케이션은 Object C 언어로 만들어야 한다.

1.2.1		자바로 만들어진 대표제품 : 웹
90년대 쯤 인터넷의 활성화 되기 시작하면서 수 많은 정보들이 월드와이드웹(World Wide Web, WWW)을 통해서 만들어지기 시작하였다. http 프로토콜을 통해서 전송되는 정보는 웹사이트를 통해 유통되었고, 웹사이트는  php, cgi, servlet/jsp 와 같은 기술들을 이용하여 구축되었다.  php, cgi, servlet/jsp와 같은 기술들은 사용자가 웹페이지를 호출하는 정보를 받아서 일정 형태로 가공한 뒤에 웹브라우저가 인식할 수 있는 HTML(Hyper Text Markup Language)로 내려주는 기능을 한다. 이 때 정보를 가공 할 때에는 정보를 저장하기 위하여 DB(DataBase)와 같은 기술을 사용하기도 하였다.  그림 1.4는 웹의 정보 흐름과 필요한 기술을 보여주고 있다.


[그림 1.4] 웹의 정보흐름과 필요기술


그림 1.4에서 WAS(Web Application Server)에서 구동되는 애플리케이션이 자바로 만들어졌다.  WAS라는 플랫폼은  JVM을 기반으로 구동되고 있고 앞서 살펴본 J2EE기반으로 개발된 자바 프로그램이 실행될 수 있다.  J2EE기반으로 웹 애플리케이션은 한번 작성된 코드로도 여러 플랫폼에서 실행이 가능하였고 JSP와 같은 스크립트가 지원되어 쉽게 웹 애플리케이셔을 만들 수 있어 많은 개발자들로 부터 각광을 받게 되었다.  특히 한국에서는 90년대 후반부터 정부 주도로 많은 자바 개발인력이 배출되어 많은 현장에서 자바를 기반으로 한 웹 시스템들이 많이 개발 되었고, 다년간 성능과 안정성에 대한 검증이 충분히 이루어져서 지금도 현장에서는 웹기반 시스템을 개발할 때 압도적으로 많이 사용하고 있다.

1.2.2	자바로 만들어진 대표제품 : 안드로이드
안드로이드는 구글에서 만들고 관리하는 스마트폰 운영체제이다. 삼성, LG와 같은 스마트폰 제조사들인 대부분 안드로이드를 운영체제로 탑재하고 있다. 글을 쓰는 지금 시점에서는 타이젠, 우분투와 같은 리눅스 기반의 운영체제도 나오고 있다. 안드로이드는 아이폰OS와 다르게 자바 기반으로 만들어졌기 때문에 풍부한 자바 개발자 인력들이 쉽게 앱 개발로 들어올 수 있게 해주었다. 글을 쓰는 시점에서는 아이폰 앱스토어 갯수를  안드로이드 마켓이 추월하였다.  안드로이드는 Java VM과 유사한 Dalvik 프레임워크를 기반으로 구동된다. 리눅스(Liniux)운영 체제에 Java VM을 올린 형태로 생각하면 된다. 그림 1.5는 안드로이드 프레임워크(주1)를 보여주고 있다.

자바를 설치한다는 의미는 사용하는 운영체제에 JDK 또는 JRE 를 설치하는 것을 의미한다.  JRE(Java Runtime Environment)는 자바 프로그램을 실행하기 위한 플랫폼만 가지고 있다. .java 소스파일을 컴파일 하기 위해서는 JDK(Java Development Kit) 을 설치해야 한다.  자바 플랫폼의 주요 기능들을 살펴보자.

- 개발 툴 : 자바 플랫폼을 설치하게 되면 컴파일, 구동, 모니터링, 디버깅, 문서화에 필요한 기능들이 포함된다.
- API(Application Programming Interface) : 애플리케이션을 만들 때 유용한 클래스, 인터페이스 모음을 제공한다. 기본적인 클래스부터 XML 만들고, 데이터베이스를 연결하는 것과 같은 고급 라이브러리도 제공한다.
- 배포 툴 : 애플리케이션을 최종  사용자에게 배포하기 위한 Java Web Start 와  Java Plug-In 도 제공한다.
- 사용자 인터페이스 툴킷 : Swing이나 Java 2D 툴킷과 같은 그래픽 유저 인터페이스(GUI)를 제공한다.
- 통합 라이브러리 : Java IDL API, JNDI등과 같은 외부 기술과 자바를 통합하기 위한 라이브러리도 같이 제공한다.

1.4		첫번째 자바 프로그램
JDK를 설치하면 컴파일을 수행하고 만들어진 .class 파일을 실행 할 수 있다. 하지만 JDK는 소스코드를 편집하기 위한 별도의 툴을 제공하고 있지 않기 때문에 우리는 운영체제에서 제공되는 텍스트 편집기 또는 오픈소스 편집기를 이용해서 자바 코드를 작성할 수 있다. 필자의 경우 윈도우 환경에서는 무료로 사용할 수 있는 notepad++을 텍스트 편집기로 주로 사용하고 있다.  또는 윈도우를 사용하는 독자들은 메모장을 이용해서 소스코드를 편집할 수 있다.  notepad++을 http://notepad-plus-plus.org/ 에서 최신버전을 다운로드 받을 수 있다.  설치과정은 매우 쉽기 때문에 별도로 설명하지 않겠다. 윈도우에서 기본으로 제공하는 메모장을 사용하고자 하는 독자는 [실행 -> 프로그램 -> 보조프로그램 -> 메모장]을 클릭하여 메모장을 실행하면 된다.

// 안녕하세요 여기는 주석입니다.

대소문자를 구분하여 조심히 입력해야 한다. 모두 입력하였다면 우리의 첫번째 프로그램의 소스코드는 모두 완성된 것이다. 메모장 또는 notepad++ 프로그램의 [메뉴 -> 편집 -> 다른이름으로 저장..]을 눌러서 파일명에 HelloWorld.java 를 입력한 뒤에 저장하자. 그림 1.15를 참조하여 C:\ 디렉토리 밑에 저장하도록 하자.

1.5		일반적인 자바코드
지금부터는 1.4 에서 작성하여 실행시켜 보았던 HelloWorld.java의 소스코드를 분석해서 자바 프로그램이 어떻게 동작하고 무엇으로 구성되어 있는지 살펴보자.

[예제 1.2] HelloWorld.java 소스코드
// 안녕하세요 여기는 주석입니다.
/* 이 프로그램은 “Hello World!”라는 글자를 출력하는 프로그램 입니다. */
class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello World!");
}
}

예제 1.2는 예제 1.1과 같은 소스 코드이다. 이 소스코드는 크게 3가지 부분으로 구성이 되어 있다. 주석(comment)와  클래스(class) 라는 정의,  메인(main) 메소드라고 부르는 부분이다. 각 부분을 좀 더 자세히 살펴보도록 하자.

1.5.1 주석
주석(comment)라고 부르는 부분은 컴파일러에 의해서 무시되는 부분이다. 컴파일러에게 이 부분을 무시하라고 나타내는 일종의 기호이다. 이곳에 사람이 인식할 수 있는 프로그램의 설명과 같은 내용들을 기재할 수 있다. 자바에서는 크게 3가지 주석의 종류를 제공한다.

/* 주석 내용을 기재  */
컴파일러는 /* 과  */ 사이의 모든 내용을 무시하게 된다.

/** 문서화 */
이 주석은 도큐멘트를 나타내는 주석이다. 도큐멘트는 흔히 자바 독(java doc) 문서라고 부른다. 이 주석 내에 메소드나 클래스에 대한 설명을 기재하고 javadoc 툴로 컴파일 하게 되면 자동으로 프로그램에 대한 설명을 담고 있는 API 문서가 생성된다. javac 컴파일러는 이 종류의 주석도 무시한다.

// 주석 내용을 기재
컴파일러는 // 이 개재된 라인의 내용만 무시한다.

1.5.2	 class 정의
예제 1.3에서 볼드체로 표시된 부분이 HelloWorld의 클래스 정의 부분이다.

[예제 1.3] HelloWorld의 class 정의
class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello World!");
}
}

모든 자바 소스코드는 class로 감싸인 블록 내에서 기재해야 한다. 클래스를 정의하는 방법은 아주 간단하다.  다음과 같이 class 라는 키워드 뒤에 사용하고자 하는 클래스명을 넣으면 된다.

class 클래스명 {

}

예를 들어 HelloJava 라는 클래스를 정의하고자 하는 경우에는  다음과 같이 class 키워드 뒤에 HelloJava 라고 정의하고 중괄호블록({, }) 열고 닫으면 된다.

class HelloJava{

}

소스코드는 반드시 class의 중괄호 내에 기재해야 한다. 그리고 중괄호는 반드시 쌍을 맞추어 열고 닫아야 한다. 즉 { 중괄호로 시작했다면 반드시 } 중괄호로 닫아야 한다. 자세한 내용은 6장에서 좀  더 자세히 다룰 것이다.

1.5.3	 main 메소드
다음은 main 메소드에 대해 알아보자. 예제 1.4에서 볼드체로 표시된 부분이 main 메소드 이다.

[예제 1.5] HelloWorld의 main 메소드
class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello World!");
}
}

자바 애플리케이션 프로그램은 반드시 main 메소드가 있어야 실행이 된다. main 메소드는 public static void main(String[] args) 로 사용해야 한다.  여기서 public과 static 은 순서를 바꾸어 사용해도 된다. public static 또는 static pulbic과 같이 사용해도 된다. public과 static 키워드는 각각 의미가 있지만 여기서는 다루지 않고 7.5 장에서 다룰 것이다.  또 하나 눈여겨 볼 부부은 main(String[] args)이다 String[] args는 main 메소드로 전달되는 값이다. 이 값은 프로그램을 실행할 때 자바 가상 머신에 의해서 자동적으로 전달된다.

java HelloWorld hi hello

이렇게 프로그램을 실행하게 되면 main 메소드의 인수인 args에는 “hi”와 “hello”라는 문자열이 전달된다. main 메소드의 블럭 내에서 이를 꺼내어 사용할 수 있다.

1.5.4	 API 문서
다음으로는 API 문서에 대해 살펴보자. 앞에서 Java VM은 두 가지 부분으로 이루어진다고 했다. 프로그래을 실행하는 플랫폼과 프로그램에 필요한 유용한 모음을 뫃아 놓은 API이다. API문서는 이 API들에 대한 설명이다. Java의 기본적인 API문서는 프로그래밍 하면서 살펴보아야할 필수적인 문서이다. API 내용들은 매우 방대하지만 우리는 현장에서 사용하는 필수적인 API를 이 책에서 자세히 공부할 것이다. 이번 장에서는 API문서의 구조와 API 문서를 보는 방법을 살펴볼 것이다. 그림 1.16은 Java VM에서 제공하는 기본 API문서이다. 이 문서는 JDK가 설치된 디렉토리 밑의 doc 디렉토리에 index.html을 브라우저로 열면 볼 수도 있고 오라클 사에서 제공하는 http://docs.oracle.com/javase/7/docs/api/ 문서를 통해서 우리가 설치한 Java SE 7 버전의 API문서를 조회할 수 있다. 이 주소는 오라클사의 정책을 통해 변경될 수 도 있기 때문에 구글이나 다음과 같은 검색엔진을 활용하여 찾아보는 것도 좋은 방법이다.

1.6		자바용 IDE 사용법
우리가 앞에서 HelloWorld 프로그램을 만들때 메모장과 같은 텍스트편집기로 자바소스코드를 만들고 나서 도스창을 열어서 javac 컴파일러로 컴파일 한 뒤에 java 명령어를 실행하여 HelloWorld 프로그램을 실행하였다. 총 3 단계를 거쳐서 프로그램을 만들어 실행하였다. 간단한 프로그램은 상관 없지만 매우 긴 소스코드를 가진 프로그램을 만들어야 하거나 여러사람이 같은 프로그램을 만들어야 한다거나 하는 복잡한 작업에는 생산성이 떨어진다. 그래서 실제 SI(System Integration) 현장이나 SM(System maintainence)현장과 같이 프로그램을 다루는 현장에서 실무 개발자들은 IDE(Integrated Development Envrionment)라고 부르는 통합 개발환경을 사용한다. 현재 국내에서 가장 많이 사용하는 툴은 이클립스(eclipse)를 사용한다. 이클립스 이외에도 오라클에서 제공하는 넷빈IDE(NetBean IDE)도 있고, 인텔리J(IntelliJ)와 같은 IDE들이 있다. 하지만 여기서는 국내에서 가장 많이 사용하는 이클립스 IDE만 설명하도록 하겠다.

1.6.1		이클립스 설치방법
이클립스는 IBM에서 제공하는 오픈소스이다. 즉 누구나 무료로 사용할 수 있다. 버전도 계속 업데이트 되고 있고, 가장 많은 자바 개발자들이 이용하는 통합 개발자 환경이다. 이클립스 IDE를 다운 받기 위해서는 이클립스 다운로드 페이지(eclipse.org/downloads/)에서 다운을 받자. 이곳에서는 다양한 이클립스 버전들이 존재한다. 이클립스로는 자바 프로그래밍 뿐만 아니라 C++프로그램, 모바일 프로그램을 만들 수 도 있다. 각 목적에 맞는 이클립스가 제공되고 있기 때문에 아무것이나 다운로드 받으면 안된다. 여기서는 가장 기본적인 이클립스인 Eclipse Classic을 이용하자. 그럼 이클립스 다운로드 페이지에서 Eclipse Classic에서 각자 운영체제에 맞는 이클립스 버전을 다운로드 받아서 각자의 컴퓨터에 설치해보자.

설치하는 과정은 그다지 어렵지 않다. eclipse.zip파일을 다운로드 받은 뒤에 이를 해당 디렉토리에 풀면 설치는 모두 완료된다. 압축이 풀려진 디렉토리 내에서 윈도우의 경우라면  eclipse.exe, 맥 OSX나 리눅스라면 eclipse 파일을 더블클릭하여 프로그램을 구동하여 보자. 처음 구동하면 이클립스를 더블클릭하여 실행하게 되면 select a workspace를 묻는 창이 뜬다. 이 때에 작업할 소스코드를 저장할 하드디스크 상의 디렉토리를 설정하면 된다. 이 단계에서 기본 디렉토리를 사용하기 위해서는 ok 버튼을 누르고 넘어가자. 이클립스를 처음에 구동시키면 Welcome to Eclipse 화면이 뜨는데 Welcom 탭의 x 버튼을 눌러서 창을 닫자. 그림 1.18는 실행된 이클립스 환경이다.

1.7   자주 발생하는 에러와 해결방법
이전 장에서 정확하게 따라했다면 에러는 볼 수 없었을 것이다. 하지만 어떤 독자들은 예기치 못한 에러에 직면했을 수도 있다. 이번 장에서는 우리가 자바 프로그램을 만들다 보면 자주 접할 수 있는 에러들과 에러에 대한 해결 방법을 배울 것이다.
소스에서 에러를 발견하는 작업을 디버깅(Debugging)이라고 하고 소스에서 발생된 에러는 버그(bug)라고 부른다. 프로그램을 작성하다 보면 소스 이외에도 다양한 이유로 에러가 발생하여 프로그램이 정상적으로 실행되지 않는다. 처음으로 자바 프로그램을 만들면서 기본적으로 직면할 수 있는 에러들을 종류 별로 구분한다면 컴파일러나 프로그램 실행환경등에서 발생하는 에러가 있을 수 있고 소스코드에서 규칙을 따르지 발생하는 에러가 있을 것이다.

환경설정 측면의 에러
윈도우에서 자바를 작성해본 일부 독자들은 컴파일 시에 다음과 같은 메시지를 본 독자가 있을 것이다.

C:\>javac HelloWorld.java
'javac' is not recognized as an internal or external command, operable program or batch file

위 메시지는 컴파일러가 정상적으로 설치되지 않았을 경우에 발생하는 에러이다. 즉 javac 명령어를 인식하지 못해서 발생하는 에러이다. 이 경우에는 두 가지 해결방법이 있는데 첫번째는 ‘자바설치디렉토리\bin\javac’로 실행하는 방법이다. 예를 들어

C:\Program Files\Java\jdk1.7.0_17\bin\javac HelloWorld.java

로 실행할 수 있다. 두번째는 1.3.2 처럼 환경변수를 제대로 설정 하였는지 확인해보면 된다.

컴파일 시점의 에러
다음은 컴파일 시에 자주 볼 수 있는 에러에 대해 설명하겠다. 가장 흔하게 소스코드를 타이핑하다 발견할 수 있는 에러는 다음과 같이 ‘;’이 필요하다는 에러이다.

HelloWorld.java:3: ';' expected
System.out.println("Hello World!!!")
^
1 error

아주 친절하게도 컴파일러는 ; 이 빠졌다고 알려주고 있다. 이 경우에는 HelloWorld.java 3 번째 줄의 끝에 ; 를 추가하면 된다. 또한 다음과 같이 System.out.println 대신에 System.ut.println이라고 오타를 친 뒤에 컴파일 하게 되면 컴파일러는 정확하게 무엇이 잘못된 부분인지 예측해서 알려준다.

HelloWorld.java:3: cannot find symbol
symbol  : variable ut
location: class java.lang.System
System.ut.println("Hello World!!!");
^
1 error

만약 몇 개의 에러가 동시에 발생하는 경우에 한 개씩 에러를 해결하면서 컴파일 해가면 에러는 잡힐 수 있다.

실행 시점의 에러
다음으로 윈도우에서 자바를 실행할 때 다음과 같은 에러를 본 독자들도 있을 것이다. 예를 들어

C:\>java HelloWorld
Exception in thread "main" java.lang.NoClassDefFoundError: HelloWorld
Caused by: java.lang.ClassNotFoundException: HelloWorld
at java.net.URLClassLoader$1.run(URLClassLoader.java:202)
at java.security.AccessController.doPrivileged(Native Method)
at java.net.URLClassLoader.findClass(URLClassLoader.java:190)
at java.lang.ClassLoader.loadClass(ClassLoader.java:306)
at sun.misc.Launcher$AppClassLoader.loadClass(Launcher.java:301)
at java.lang.ClassLoader.loadClass(ClassLoader.java:247)

이 메시지는 java를 실행할 때 HelloWorld.class 가 실행한 디렉토리에 존재하는지 확인해봐야 한다. 이 에러는 HelloWorld.class 바이트 파일을 java VM이 인식하지 못해서 생기는 문제이다. 윈도우에서 dir 명령어로 HelloWorld.java 파일과 HelloWorld.class 파일이 같이 있는지 확인해보고 HelloWorld.java 가 제대로 컴파일 되었는지 확인하기 바란다. 지금까지 컴파일 시점에 발생할 수 있는 에러에 대해 공부하였다. 사실 제일 잡기 힘든 에러는 컴파일 시점의 버그보다는 런타임 시점의 버그이다. 런타임 시점의 버그를 잡기 위해서는 로그를 찍어서 확인해봐야 한다.

에러를 잡는 것은 프로그래밍에서 가장 중요한 일 중에 하나이다. 실제 프로그래밍할 때 코드를 작성하는 시간보다 에러를 잡는 것이 더 많은 시간이 소요될 수도 있다. 에러를 많이 접해보고, 에러를 많이 잡아본 경험도 프로그래밍 실력 중에 한 가지이기 때문에 직접 해결해보려고 노력하는 것이 실력향상의 지름길이다.


1.8 이 책의 예제를 실행하는 방법
이 책에서 노트패드와 같은 일반 편집기를 이용해서 .java파일을 작성한 뒤에 도스창과 같은 커맨드라인창에서 javac 명령어와 java 명령어로 직접 컴파일하고 파일을 실행하는 방법으로 예제를 실행할 수도 있고, 이클립스와 같은 IDE를 통해서 예제를 실행할 수 있는 방법이 있다. 이 책에서 예제를 실행하는 방법에 대한 설명은 노트패드와 같은 일반 편집기를 이용해서 .java파일을 작성하고 커맨드라인창에서 javac, java 명령어로 직접 컴파일 하는 것을 기준으로 설명하고 있다. 예제 소스에 저장 해야 할 파일명, 컴파일 방법, 실행 방법을 포함하고 있다.

하지만 이클립스를 사용하는 독자들은 예제 파일을 돌려보기 위해서 별도의 패키지를 지정하는 것이 더 편리하다. 패키지는 .java 파일의 첫 라인에 다음과 같이 지정할 수 있다.

패키지는 클래스들을 일종의 단일 네임스페이스공간으로 묶어줄 수 있기 때문이다. 예를 들어 ex1_1 패키지에는 여러 개의 .java 파일들이 들어갈 수 있다. 패키지로 파일들을 묶어주면 단일 기능 또는 단일 목적을 위한 파일들을 별도로 구분할 수 있다. 이클립스 IDE를 사용해서 하나의 프로젝트에서 모든 예제를 실행시키기 위해서 패키지를 지정하는 것이 편리하다. 패키지의 이름은 임의대로 지정할 수 있지만, 이 책에서는 예제 1.1은 ex1_1과 같은 패키지를 지정하였고 예제 2.1은 ex2_1과 같이 지정하여 제공하고 있다. 저자의 블로그(blog.pizzastudio.org)에서 이클립스용으로 포팅된 예제와 일반 예제 파일을 받을 수 있다.

이클립스용으로 포팅된 예제를 다운받고 압축을 푼 뒤에 이클립스의 [File > Import] 메뉴를 이용해서 [General > Existing Projects into Workspace]로 예제 디렉토리를 선택하면 된다.


1.9	요약
1장에서 자바 프로그램에 대해 기초적인 부분을 살펴보았다. 자바가 무엇인지에 대해 알아보았고, 자바가 탄생한 배경과 어떻게 개정 되었는지에 대해 간단히 알아보았다. 우리가 자바를 이용해서 웹사이트를 구축할 수 있었고, 안드로이드 프로그램도 만들 수 있다는 사실을 배웠다. 이 외에도 데이터베이스, 검색엔진, 대용량 처리 플랫폼과 같은 수 많은 소프트웨어 제품들이 자바로 만들어졌고, 자바는 지구상에서 가장 많은 개발자를 보유한 언어이기도 하다.
자바 언어를 이용해서 우리의 최초의 프로그램인 HelloWorld을 간단하게 만들어 보았다. HelloWorld 프로그램은 보잘 것 없어 보이지만 우리가 이 책을 모두 공부했을 시점에는 보다 멋진 그래픽 UI를 가진 프로그램과 네트워크를 이용한 고급 프로그래밍도 할 수 있을 것이다. 또한 이 책을 공부하고 바로 안드로이드 프로그래밍을 시작해도 좋다. HelloWorld 프로그램을 통해서 자바 프로그램의 기본 골격을 공부하였다. 이클립스와 같은 통합개발환경(IDE)를 사용하면 훨씬 더 쉽게 자바 프로그램을 작성하고 실행할 수 있었다. 이러한 이유로 현장에서는 거의 대부분 통합개발환경을 사용한다. 지금까지 따라오신 독자들은 이제 자바 프로그램을 배울 수 있는 모두 자격을 갖추신 셈이다. 다음 장 부터는 자바의 기본적인 측면부터 차근차근 공부할 것이다. 계속 초심을 잃지 않고 끈기 있게 따라온다면 이 책을 모두 마스터할 수 있을 것이다.
