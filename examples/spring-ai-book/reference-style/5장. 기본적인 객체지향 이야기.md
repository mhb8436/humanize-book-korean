5. 객체지향의 개념
자바는 객체지향(Objected-Oriented)언어이다.  객체지향 프로그래밍 패러다임은 1950년대 후반에 등장하였다. 하지만 C++이나 Object-C에 Java에 이르어서 본격적으로 사용되었고 현재의 프로그래밍 패러다임을 주도하고 있다. 지금 부터는 자바의 핵심 개념인 객체지향에 대한 기본 개념을 공부해보자.

5.1 객체지향언어의 역사
객체지향 프로그래밍 방식은 1960년대에 MIT에서 처음 만들었다. 객체지향언어로서 공식적으로 인정받는 언어는 1960년대에 등장한 Simula 였다. Simula는 클래스와 인스턴스 또는 객체와 같은 개념을 도입하였다. 또한 자동 가비지 컬렉션의 개념도 도입하였다. Simula의 아이디어는 이후 Object Pascal과 C++ 등에 많은 영향을 미쳤다.
1970년대 Xerox에서 Smalltalk언어가 개발되었고  Samlltalk에 객체지향 개념이 도입되었다. Samlltalk를 만든 개발자들은 Simula와 틀리게 동적인 객체지향 개념을 도입하였고 1981년 발표하였다. Smalltalk의 개념은 Lisp 개발자들에게 영향을 미쳤고 Lisp에 다중상속등과 같은 일부 객체지향의 개념을 첨가한 Common Lisp으로 만들어졌다.
1980년대에는 순수 객체 지향언어를 표방한 Eiffel 언어가 발표되었다. 1990년대에 기술적으로 대중에게 인정받아 널리 사용하게된 언어는 바로 C++ 및 Delphi이다. 1990년대 이후로 대중적으로 그래픽 사용자 인터페이스(GUI)가 인기가 높아짐에 따라 객체지향 프로그래밍 기술이 각광받게 되었다.
객체 지향언어 중에서 상업적으로 가장 큰 성공을 거둔 언어는 선마이크로시스템즈 사의 자바(JAVA)와 마이크로소프트 사의 .NET의 C#일 것이다.  C# 은 교차 언어 상속과 같은 기능으로 다른 언어에서 클래스의 사용도 가능하며 자바의 경우 자바 가상 머신을 통해서 모든 운영 체제에서 실행이 가능하다.


5.2	객체지향의 개념과 구성요소
객체 지향 프로그래밍에서 객체는 언어를 빗대어 설명하자면 명사와 동사를 합친 형태라고 할 수 있다. 명사는 앞서 배웠던 기본형 자료형의 변수, 문자열 등이고 동사는 잠깐 살펴봤던 메서드이다. 동사는 주로 객체의 행위나 거동들을 나타낸다.
객체지향 프로그래밍은 순차적으로 작업 목록이 수행되는 기존 절차 지향 프로그래밍 방식과 다르게 객체가 서로 상호작용하는 프로그래밍 모델이라고 볼 수 있다. 객체는 자신이 가지고 있는 데이에 대한 책임을 가지고 있으며, 다른 객체에 메시지를 보낼 수 있고 수신도 할 수 있는 독립적인 단위이다.
절차 지향 프로그래밍에서는 프로그램은 매우 긴 함수의 목록을 가질 수 있다. 복잡한 프로그램을 작성해야 하는 겅우에는 일부 데이터를 수정하여 버그가 발생할 경우에는 영향 범위가 광범위 해질 수 있다. 반면 객체지향 프로그래밍에서는 다른 객체에서 그 객체의 데이터에는 접근할 수 없다. 객체가 가지고 있는 데이터에 접근하기 위해서는 별도의 함수(메서드)를 통해서만 접근할 수 있다. 데이터와 데이터를 접근하고 관리하기 위한 메서드를 가지고 있는 세트를 객체라고 할 수 있다.
객체는 실세계 모델을 프로그램으로 옮긴 것이라고 볼 수 있다.  은행 계좌를 예를 들어보자. 실세계에서 개인들이 몇 개씩의 은행 계좌를 가지고 있듯이 각각의 은행 계좌가 있을 수 있듯이 객체는 프로그램 내에서 동일한 타입으로 여러 개가 존재할 수 있다. 객체는 데이터를 가지고 특정 작업을 수행하기 위한 간단하고 표준적인 방법을 제공해준다. 그래서 기능 또는 데이터를 변경하기 위해서 전체 프로그램을 손대지 않고 간단한 변경이 가능하다.  이러한 기능 덕분에 프로젝트에서 프로그램 또는 모듈의 재사용이 훨씬 편해졌다.

5.3		객체
객체는 객체지향언어의  핵심 요소이다. 우리가 반드시 이해하고 넘어가야 할 개념이다. 앞서 간략히 언급하였다시피 객체는 현실 세계를 모델링한 개념이고 객체 내에는 명사형의 상태와 동사형의 메서드가 존재한다고 하였다.
예를 들어 사람이라는 객체가 있다고 가정하자. 실 세계에서 사람은 이름과 머리색깔, 코의 높낮이, IQ와 같은 상태가 있을 것이고, 말을하고, 걷고, 뛰고, 일을하고, 음식을 먹는 등의 행위등으로 묘사를 할 수 있을 것이다. 물론 이 상태나 거동들이 모든 사람을 표현하기 에는 충분치는 않다. 다른 예를 들어보자. 자동차를 객체로 묘사해보자. 물론 자동차의 모든 속성을 전부 객체로 묘사할 수는 없다.

우리가 공부해야하는 객체지향 프로그래밍의 객체는 개념적으로 지금까지 연습하였던 실세계의 객체와 유사하다.  객체는 상태와 행위를 가지고 있다.  객체의 상태는 필드(fileld)에 저장이 되고 행위는 메서드(method)로 표현된다. C 또는 C++과 같은 일부언어에서는 메서드를 함수라고 하기도 한다. 메서드의 주요 역할은 객체의 상태를 관리하고 외부에 객체의 상태를 제공해주거나 외부의 요소에 의해 객체의 상태를 수정하는 역할도 수행한다.  객체의 내부상태를 외부에서 보호하는 메서드의 이러한 역할을 데이터 캡슐화(data encapsulation) 이라고 한다.

앞서 공부하였던 실세계의 컴퓨터 객체를 객체지향 프로그래밍의 객체로 생각해보자. 객체의 필드에는 CPU, 메모리, 하드디스크, 화면크기 등이 올수 있고, 메서드에는 컴퓨터의 현재 상태를 볼 수 있는 메서드와 CPU 또는 메모리, 하드디스크와 같은 필드에 접근할 수 있는 메서드가 올수 있다.  이 객체의 메모리는 현재 2GB 이지만 메모리의 상태를 변경할 수 있는 메서드를 이용해서 다른 객체에서 메모리를 4GB로 업데이트 할 수 있다.  이렇게 객체를 사용하게 되면 몇 가지 좋은 점이 있다.

기존에 만들어진 객체의 경우에는 프로그램에서 다시 사용할 수 있다. 예를 들어 다른 개발자나 다른 곳에서 만들어진 객체를 우리가 만든 프로그램에서 쉽게 가져다 사용할 수 있다.
상태 정보는 객체의 메소드를 통해서만 접근할 수 있어 내부의 구현은 뵈부 객체에서 보기에 숨겨져 있기 때문에 상태 정보는 안전하게 보관될 수 있다. 객체의 소스코드는 다른 객체와 별도로 작성되기 때문에 다른 객체의 소스코드 수정에 영향이 적다. 생성된 객체는 손쉽게 시스템 포팅될 수 있다.
특정 객체에서 문제가 발생 할 경우에는 객체만 제거하면 되기 때문에 쉽게 추가하였다 삭제하였다 할 수 있다. 또한 문제가 생긴 객체를 손쉽게 다른 객체로 교체할 수 있다.

5.4		클래스
우리 주변에서 찾아볼 수 있는 대부분의 공산품 류는 하나의 제품을 여러 개 찍어서 소비자에게 판매한다. 예를 들어 현대 자동차의 제네시스라는  자동차는 국내에만 수십대가 볼 수 있다. 공장에서는 제네시스라는 제품을 만들어서 계속 판매한다. 자동차 뿐만 아니라 노트북도 마찬가지이다. 맥북 에어 11인치, 4기가 메모리, 1.6GHz CPU를 가진 제품은 전 세계에 수십만대 또는 수백 만대가 있을 것이다.   맥북에어 11인치 모델에 대한 설계도를 이용해서 공장에서는 제품을 만들어 내고 판매한다.

객체지향 프로그래밍에서 객체와 클래스의 관계는 이와 비슷하다.  클래스는 설계도와 공장에 비유할 수 있고 객체는 공장에서 만들어진 제품 한 개에 비유될 수 있다. 클래스는 객체를 만들어 내기 위한 일정한 틀이고 객체는 틀에서 만들어진 제품이다. 그림 5.1은 이러한 개념을 표현하고 있다.



[그림 5.1] 클래스와 객체의 관계



그림 5.1과 같이 객체지향의 세계에서 맥북에어는 클래스에서 만들어진 객체이며 이는 클래스의 인스턴스라고 부르기도 한다.   그럼 실제 코드 상에서 클래스와 객체가 무엇인지 확인해보자. 예제 5.1은 실제 자바 코드이다.

[예제 5.1]
// macbookair .java
public class macbookair {
int processor = 2;
int memory = 4 ;
int hardDisk = 128;
public int getProcessor() {
return processor;
}
public void setProcessor(int processor) {
this.processor = processor;
}
public int getMemory() {
return memory;
}
public void setMemory(int memory) {
this.memory = memory;
}
public int getHardDisk() {
return hardDisk;
}
public void setHardDisk(int hardDisk) {
this.hardDisk = hardDisk;
}
}

// applefactory .java
public class applefactory {
public static void main(String[] args){
macbookair a = new macbookair();
a.setProcessor(3);
a.setMemory(8);
a.setHardDisk(256);
macbookair b = new macbookair();
b.setProcessor(1);
b.setMemory(4);
b.setHardDisk(64);
macbookair c = new macbookair();
c.setProcessor(4);
c.setMemory(16);
c.setHardDisk(512);

System.out.println("a 맥북에어 CPU:" + a.getProcessor() +
" MEMORY::" + a.getMemory() + " HARDDISK:" +  a.getHardDisk());
System.out.println("b 맥북에어 CPU:" + b.getProcessor() +
" MEMORY::" + b.getMemory() + " HARDDISK:" +  b.getHardDisk());
System.out.println("c 맥북에어 CPU:" + c.getProcessor() +
" MEMORY::" + c.getMemory() + " HARDDISK:" +  c.getHardDisk());
}
}




예제 5.1에서 macbookair 클래스는 macbookair를 찍어내기 위한 설계도 또는 틀이다.  applefactory 클래스를 살펴보자.  다음과 같이 new 키워드로 새로운 객체를 생성하였다.

macbookair a = new macbookair();

이 키워드의 의미는 macbookair 클래스를 이용해서 a 라는 이름의 macbookair 객체를 생성하라는 의미이다.  그리고 나서 다음과 같이 a 객체의 메서드를 이용해서 a 객체의 상태값을 변경하였다.

a.setProcessor(3);
a.setMemory(8);
a.setHardDisk(256);

setProcessor() 는  macbookair 클래스에서 있는 메서드이다. setProcessor 메서드는 int 형매개변수를 받는다. 여기서는 3이란 숫자를 메서드의 매개변수로 전달하였다. macbookair 클래스의 setProcessor 메서드의 내용을 보면 this.processor = processor; 와 같이 매개변수로 받은 processor 변수를 this.processor 변수에 할당하였다. this 자바에서 제공하는 키워드이고 의미는 현재 상태의 객체를 반환하라는 의미이다. 이 문자의 의미는 해당 객체의 processor 변수에 매개변수로 받은 processor 값을 할당하라는 의미이다. 그럼 여기서 3을 setProcessor 메서드의 매개변수로 전달하였으므로 a 객체의 processor 라는 변수에는 3의 값이 설정된다.
a 객체의 setProcessor 메서드를 호출하기 위해서는 a.setProcessor  와 같이 “.”을 이용해서 호출하면 된다.

객체이름.메서드명

a객체의 setMemory, setHardDisk 메서드도 동일한 방식으로 모두 변경하였다.  b 객체와 c 객체를 생성하고 각각의 객체에 다른 값들을 설정하였다. a, b, c 객체 내에 값을 확인하기 위해서 System.out.println 메서드로 확인하였다.

System.out.println("a 맥북에어 CPU:" + a.getProcessor() +

“a 맥북에어 CPU”는 문자열이고  a.Progressor() 메서드는 int  형의 데이터를 반환한다.  macbookair 클래스의 getProcessor 메서드를 살펴보자.

}

메서드 내부에서는 return processor;와 같이 processor 변수에 저장된 값을 반환한다. a 객체에는 앞서  setProcessor 메서드를 이용해서 3이란 값을 prcoessor 변수에 할당하였기 때문에 getProcessor 메서드에서는 숫자 3 이 반환될 것이다.  메모리, 하드디스크 사이즈를 나타내는 getMemory(), getHardDisk() 메서드도 getProcessor() 메서드와 동일하다.

지금까지 예제와 그림을 통해서 객체와 클래스의 개념을 살펴보았다. 다음에는 객체지향언어에서 중요한 개념인 상속에 대해 배워보도록 하자.

5.5		상속(Inheritance)
실제 상속이란 의미는 다음 대에게 무엇을 물려주는 행위를 말하는데  객체지향 프로그래밍에서  상속은 어떤 특성들을 물려받는 다는 의미이다. 어떤 특성이라 함은  공통적인 특성을 의미한다. 예를 들어  사람의 경우 부모는 자식에게 유전자를 물려주게 되는데 유전자의 영향에 따라 부모의 특성이 자손에게 전달된다.
앞서 클래스와 객체를 공부할 때 다루었던 컴퓨터에 대해 생각해보자.  컴퓨터는 노트북, 데스크톱, 서버, 워크스테이션등 종류가  다양하다. 이들은 공통된 특성도 있고 각자만의 개별적인 특성도 있다. 공통적인 특성으로는  CPU라고 불리는 프로세서를 가지고 있고, 메모리와 하드디스크와 네트워크 카드도 있다. 하지만 데스크탑의 경우에는 모니터는 별도로 존재하고 노트북은 모니터가 같이 있는 형태이며, 크기가 서로 다르고 모양도 다르다.  이런 서로 다른 특성들이 개별적인 특성이고 프로세서, 메모리, 하드디스크 등이 공통적인 특성이다.  그림 5.2는 컴퓨터의 공통 특성과 개별 특성들을 보여주고 있다.


[그림 5.2] 컴퓨터의 다양한 종류와 상속




컴퓨터는 공통적으로 CPU와 Memory 그리고 Hard disk 가지고 있다. 하지만 노트북, 데스크톱, 서버에 들어가는 CPU, Memory, Hard disk는 성능에 차이가 있고 노트북의 경우에는 모니터와 키보드를 가지고 있다.  객체지향 프로그래밍에서 이러한 개념을 그대로 적용할 수 있다. 객체지향 프로그래밍에서 상속은 부모 객체의 상태와  행위를 물려받는다.  예를 들어 노트북이 부모 객체인 컴퓨터를 상속 받는 경우 부모 객체의 CPU, Memory, Hard disk와 같은 상태를 그대로 물려받는다. 또한 노트북 만의 상태인 모니터, 키보드 상태를 가질 수 있다.
자바에서는 한 개의 부모에서만 상속을 받을 수 있고 다중 상속은 지원하지 않는다. 다중 상속은 여러 개의 부모에서 상속 받는 것이다. 자바에서 상속을 위해서는 extends 키워드를 사용하여 상속 받을 수 있다.

public class Notebook extends Computer{

}

자식 객체인 Notebook은 부모 객체 Computer의 상태와 메서드를 모두 물려받는다.  부모 객체의  상태는 별도로 정의하지 않고서도 Notebook 객체에서 사용 가능하기 때문에 코드를 읽기가 쉬워질 수 있지만 부모 클래스의 상태와 행위에 대한 문서화를 잘 해두어야 한다.

5.6		인터페이스(interface)
인터페이스라는 용어는 사전적인 의미로 상호작용 이라는 의미를 가지고 있다. 예를 들어 컴퓨터의 전원을 켜거나 끄는 경우 전원 스위치를 누르면( on ) 하면 내부적으로 전기신호가 제어장치에 전달 되서 컴퓨터를 부팅시키게 된다. 그 제어장치가 어떤 것인지는 자세히 알 수 없으나 그 제어장치의 어떤 기능 중에 하나는 컴퓨터의 전원을 켜고(on) 끄는(off) 기능이 있다고 하자. 실제 그 장치의 기능 중에  켜고 끄는 기능을 호출하라는 메시지는 사람이 외부에 있는 버튼을 켜거나(on) 끔(off) 함으로써 이루어 진다. 사람이 보기에는 버튼 on off 만 보이고 내부에 존재하는 제어장치의 on off의 존재 사실은 모른다.  이렇게 함으로써 컴퓨터는 내부적인 기능을 외부에서 직접 노출 시키지 않고 외부의 버튼을 이용해서 간접 노출하게 된다.
객체지향 프로그래밍에서도 인터페이스는 위와 유사한 기능을 제공한다. 인터페이스는 외부의 객체에서 내부의 메서드를 직접 개방하지 않고 인터페이스를 통해서 호출 할 수 있도록 할 수 있다. 인터페이스는 실제 내부 모습은 구현이 되지 않고 이름과 매개변수, 반환값으로만 정의 되어 있다. 자바에서 인터페이스는 다음과 같이 정의 될 수 있다. 버튼과 제어장치의 예제를 interface 와 class로 구현하여 보자.

public interface Power{ // 제어장치에서 제공하는 전원 on off 인터페이스
void changeOn();
void changeOff();

}

이렇게 정의된 인터페이스를 이용하여 실제 클래스는 다음과 같이 구현한다.

class Button implements Power{ // 제어장치에서 제공한 인터페이스를 이용 Button 클래스 구현
int changeOn(){
// 내부 메서드 구현
}
int changeOff(){
// 내부 메서드 구현
}
}

인터페이스는 interface 라는 키워드를 이용하여 클래스와 유사한 방식으로 선언한다. 그런데 특이한 점은 내부를 블럭 { } 기호로 구현하지 않고 이름만 존재한다. 위의 코드 조각에서는 inteface Power는 제어장치에서 제공하는 전원 on off에 대한 사용 방법을 담고 있고 class Button에서 Power에서 제공하는 changeOn, changeOff 메서드를 구현하여  사람이 버튼을 눌렀을 때 on off 처리를 수행하는 코드를 추가하면 된다.

5.7		패키지	(Package)
객체와 인터페이스를 만들다 보면 유사한 종류나 관련이 있는 클래스나 인터페이스이가 있다. 이들을 공통적으로 구분해주기 위한 네임스페이스 역할을 하는 것이 패키지이다. 개념적으로는 컴퓨터의 폴더와 같은 개념으로 봐도 될 수 있다. 우리가 작업한 파일이나 문서들을 우리가 알기 쉽게 구분하여 놓기 위해서 폴더를 만들고 폴더 내에 이를 넣어 놓는다. 폴더라는 개념이 없다면 C:\ 또는 /home 디렉토리 밑에 수 천개의 파일들이 있다고 가정하면 파일을 찾거나 정리하기가 매우 힘들 것이다.
객체지향 프로그래밍에서도 똑같이 적용될 수 있다. 패키지라는 개념이 없다면 수 천개의 소스 파일이 한 디렉토리에 존재할 수 있고 파일들은 관리하기 힘들 것이다. 실제 패키지를 만들고 패키지가  만들어진 디렉토리로 이동하면 물리적으로 패키지명과 같은 폴더 구조가 만들어진 것을 볼 수 있을 것이다.
자바 소스코드(.java)에서 다음과 같이 패키지를 선언하면 된다.

package com.pizzastudio
class PackageTest{
// 내부 구현
}

위의 코드 조각에서 package com.pizzastudio; 라고 선언하였다. 이는 oom.pizzastudio 라는 이름으로 패키지를 정의한다는 의미하는 것이고 javac로 컴파일 한 뒤에 물리적으로 PackageTest.java 파일이 있는 위치가 가보면 com 디렉토리 밑에 pizzastudio 디렉토리가 있다.
자바에서는 우리가 프로그래밍 시에 편리하게 사용할 수 있는 굉장히 많은 라이브러리들을 제공하고 있다. 이 라이브러리들은 API(Application Programming Interface)라고 부른다. 모든 API들은 유일한 패키지 명을 가지고 있다 각 패키지 명은 유사한 목적의 기능을 제공하는 것들을 그룹핑하는 역할을 수행하고 있다.


5.6		요약
지금까지 객체지향 프로그래밍에서 제공하는 주요 개념을 살펴보았다. 객체와 클래스, 상속, 인터페이스 그리고 패키지 개념을 익혔다. 객체는 현실 세계를 반영한 개념을 담고 있으며 명사형의 상태와 동사형의 행위가 포함이 되어 있다. 클래스는 객체를 만들어내는 설계도 또는 틀이며 클래스를 이용해서 객체를 만들어 내는 코드도 살펴보았다.  인터페이스는 외부 세계와 내부와 통신하기 위한 수단 이었다. 마지막으로 패키지는 클래스와 인터페이스를 모아서 그룹화 해주는 네임스페이스 역할을 제공해주고 있다.
위의 개념들은 객체지향 프로그래밍에서 이해하고 넘어가야 한다. 여기서 다 이해하지 못하였다고 실망하지 말자. 뒤에서 클래스, 인터페이스, 상속에 대한 소스코드와 예제를 학습한 뒤에 다시 이번장을 학습하면 확실하게 이해할 수 있을 것이다.
